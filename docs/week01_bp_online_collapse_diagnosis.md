# Week 01 BP-Online Collapse Diagnosis

## Summary

During Week 01 validation of the `bp_online` baseline in the thesis fork of EOH, we found a reproducible collapse mechanism in the search loop. The system is generating different heuristic programs, and these programs are being executed by the evaluator, but many of them become behaviorally indistinguishable under the current search formulation.

The key evidence is as follows. First, valid candidates have different code hashes and different raw score magnitudes, which shows the LLM is producing genuinely different functions and the evaluator is not ignoring them. This was confirmed from `candidate_attempts.jsonl` and the post-run diagnostic script [analyze_bp_online_run.py](C:/Users/pc%20omen/Documents/experimentation/EoH/scripts/analyze_bp_online_run.py). Second, these different functions nevertheless produce the same chosen-bin sequence on the traced prefix and the same per-instance used-bin counts across all 5 search instances in the embedded `Weibull 5k` search set defined in [get_instance.py](C:/Users/pc%20omen/Documents/experimentation/EoH/eoh/src/eoh/problems/optimization/bp_online/get_instance.py). Third, once all valid candidates receive the same objective, the population-management rule in [pop_greedy.py](C:/Users/pc%20omen/Documents/experimentation/EoH/eoh/src/eoh/methods/management/pop_greedy.py) removes all but one survivor because it deduplicates by objective value only. After that, parent selection in [prob_rank.py](C:/Users/pc%20omen/Documents/experimentation/EoH/eoh/src/eoh/methods/selection/prob_rank.py) repeatedly samples that same single survivor, so later prompts are effectively conditioned on one carried-over candidate.

The mechanism behind this collapse is structural. In [run.py](C:/Users/pc%20omen/Documents/experimentation/EoH/eoh/src/eoh/problems/optimization/bp_online/run.py), the evolved heuristic only scores feasible bins based on `item` and `bins`, and the decision rule is `np.argmax(priorities)`. The diagnostic results show extreme symmetry in the search state: for the first analyzed instance, duplicate valid capacities occurred in all `5000/5000` decisions, and the chosen bin belonged to a tied-capacity group in `2513/5000` decisions. The mean duplicate-capacity fraction was about `0.99937`, and the mean chosen-capacity group size was about `1657.5`. This means many bins are indistinguishable from the heuristic’s perspective, so numerically different scoring formulas still induce the same effective bin choices.

This is therefore not a logging bug and not evidence that the evaluator skips code execution. It is a confirmed interaction between: `bp_online`’s highly symmetric state representation, deterministic `np.argmax` tie handling, and objective-only survivor deduplication. The result is a search population that collapses to one behaviorally equivalent policy even though the generated code differs textually and numerically.

For the thesis, this should be documented as a confirmed limitation of the `bp_online` EOH baseline under the current formulation. Any later system that addresses this issue, for example by using richer state, behavior-aware diversity preservation, or modified tie handling, should be treated as a new method built on top of S0 rather than as part of the baseline itself.
